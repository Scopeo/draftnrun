# Migration Deploy Strategy

The CD pipeline automatically detects the type of database migration and chooses
the correct deployment order. This avoids both downtime (scale-to-zero) when it
is not needed and schema errors when new code references columns that do not
exist yet.

## Strategies

| Strategy | When | Deploy order |
|---|---|---|
| `migrate-first` | Only additive ops (add column, create table, create index) | Run migration, then roll out new code |
| `code-first` | Only destructive ops (drop column, drop table) | Roll out new code, then run migration |
| `breaking` | Rename, type change, mixed add+drop, or manual label | Scale pods to zero, migrate, scale back up |

## How detection works

`scripts/classify_migration.py` parses the `upgrade()` function of each
migration file using Python AST and classifies every `op.*` call:

**Additive** — `op.add_column` (nullable or with `server_default`),
`op.create_table`, `op.create_index`, `op.create_unique_constraint`,
`op.create_foreign_key`, `op.create_check_constraint`, `op.create_primary_key`.

**Destructive** — `op.drop_column`, `op.drop_table`, `op.drop_index`,
`op.drop_constraint`.

**Breaking** — `op.alter_column` with `new_column_name` / `type_` /
`nullable=False`, `op.rename_table`, `op.add_column` with `nullable=False`
and no `server_default`.

**Unclassifiable** — `op.execute(...)`, `op.get_bind()`, `op.get_context()`,
`op.bulk_insert(...)`, and any unknown `op.*` call. These require a manual
`deploy_strategy` annotation (see below).

**Batch operations** — `op.batch_alter_table(...)` context managers are walked
into; their inner `batch.*` calls are classified using the same rules above.

When multiple migration files are deployed together the overall strategy is the
most restrictive: breaking > mixed (migrate-first + code-first) > single
strategy > none.

## Manual override: `deploy_strategy`

Every new migration file includes a `deploy_strategy` field (generated by the
Mako template):

```python
deploy_strategy: Union[str, None] = None
```

Set it to `"migrate-first"`, `"code-first"`, or `"breaking"` to override
auto-detection. This is **required** when the migration uses `op.execute()` or
`op.get_bind()` — CI will fail if those are present without an explicit
override.

## CI behaviour

The `detect-changes` job in `.github-ci.yml` runs the classifier on any changed
migration files and:

1. Writes a **GitHub Actions summary** table with the detected strategy.
2. **Fails the build** if there are unclassifiable ops without a manual override.
3. Emits a **warning** if the strategy is `breaking`.
4. Posts (or updates) a **PR comment** with the strategy table when migrations
   are detected. Skipped when the strategy is `none` or on `workflow_dispatch`.

## CD behaviour

The `setup` job in `build-and-deploy-k8s.yml` runs the same classifier and
passes `migration_strategy` to the `deploy` job. The deploy step branches:

- **migrate-first**: run `db-migrate` and `db-seed` pods (using the new image)
  *before* updating deployment images, then roll out all deployments.
- **code-first** / **none**: update deployment images first, wait for API
  rollout, then run `db-migrate` and `db-seed`.
- **breaking**: scale all deployments to zero, deploy + migrate, scale back up
  (same as the old `breaking-migration` label path, which still works as an
  override).

## Running locally

```bash
uv run python scripts/classify_migration.py ada_backend/database/alembic/versions/<file>.py
```

Exit codes: 0 = classified, 2 = no files given, 3 = has unclassifiable ops.
